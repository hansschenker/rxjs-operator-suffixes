# RxJS Is Three Things — Deep Dive

**Section:** Composition & Concurrency
**Insight:** RxJS is 3 things
**Lesson type:** Deep Dive
**Estimated duration:** 5 min

---

Let's spend this lesson going deep on each of RxJS's three layers — the Observable type, the operators, and the Schedulers — and on how they interact. Understanding all three and their boundaries is what lets you reason precisely about a pipeline instead of reasoning by intuition.

Start with the Observable. An Observable is a container for a push-based, lazy sequence. The laziness is essential: nothing happens until you subscribe. That's the opposite of a Promise, which starts executing immediately upon creation. When you call `interval(1000)`, no timer starts. When you call `of(1, 2, 3)`, no values are emitted. The Observable is just a description of what will happen when something subscribes to it. This laziness is what makes Observables composable: you can chain operators onto an Observable without triggering any side effects, and the whole chain only activates on subscription. The Observable is analogous to what Array is for synchronous pull sequences — a typed container with a standard interface for producing values.

The operator layer is the LINQ inheritance. Erik Meijer's core insight was that the Observable is the dual of IEnumerable: where IEnumerable represents a pull-based synchronous sequence, Observable represents a push-based, potentially asynchronous one. And because they're mathematical duals, every query operator that exists for pull sequences has a corresponding dual for push sequences. That's why you have map, filter, reduce, zip, merge — the same vocabulary that LINQ and functional programmers have used for decades, now applied to streams. The full operator set covers six families: creation (of, from, fromEvent, interval, ajax), transformation (map, scan, switchMap, mergeMap), filtering (filter, take, debounceTime, distinctUntilChanged), combination (combineLatest, merge, zip, withLatestFrom), error handling (catchError, retry, retryWhen), and timing (delay, throttleTime, bufferTime). Each family has a precise semantic that you can reason about independently.

Then there are Schedulers. This is the layer that most developers discover by accident rather than by design. A Scheduler is a data structure that decides when and where a unit of work executes. Every Scheduler implements a schedule method that receives a work function and, optionally, a delay. The built-in Schedulers map to specific parts of the JavaScript concurrency model. The queueScheduler is synchronous and runs work in a trampoline — it queues recursive work rather than pushing new frames onto the call stack, avoiding stack overflow on deeply recursive stream processing. The asapScheduler runs work on the microtask queue via Promise.resolve, so it runs before the browser has a chance to process its next macro-task. The asyncScheduler delegates to setTimeout and setInterval — this is the default for time-based operators like interval and delay. The animationFrameScheduler calls requestAnimationFrame, so emissions happen at the start of each browser frame, which keeps animated value streams in sync with the render cycle.

The key point is how these layers interact. Creation operators like interval and timer accept an optional Scheduler as their last parameter — this is how you control when ticks are produced. But you can also inject a Scheduler mid-pipeline using observeOn and subscribeOn. The observeOn operator wraps the downstream observer so that every next, error, and complete notification is delivered on the specified Scheduler. The subscribeOn operator does something subtler: it controls the Scheduler on which the subscribe call itself happens. In practice, observeOn is what you use to change the delivery context, and subscribeOn is what you'd use if you wanted to move the subscription initiation — for instance, off the main thread in an environment that supports it.

The reason Schedulers are the advanced layer is that they let you do things the first two layers can't. You can control test timing with virtual time, making marble tests that run in microseconds regardless of the time values they represent. You can keep DOM updates aligned with the render cycle without manual requestAnimationFrame calls. You can reason about the relative ordering of synchronous and asynchronous work in a single pipeline. Operators describe the what. Schedulers control the when. Both are necessary, and the interaction between them is where advanced RxJS mastery lives.

what is functional programming
Based on the presentation by Józef Flakus in this video (08:45-12:35), functional programming is a paradigm centered around five key pillars that allow for more predictable and testable code:

Pure Functions: These functions have no side effects (such as logging, database access, or file system interaction). Given the same set of parameters, they will always return the same output, ensuring predictability (08:52-09:52).
Referential Transparency: This is closely tied to pure functions, meaning you can replace a function call with its returned value without changing the program's behavior (09:55-10:25).
Immutability: Instead of modifying existing data structures, you work with immutable structures and create copies when changes are needed, which helps avoid common errors (10:27-11:05).
Functions as First-Class Citizens: You treat functions like any other variable—they can be assigned to constants, passed as arguments to other functions, or returned from them (11:07-11:45).
Recursion: In pure functional programming, traditional loops (like for loops) are avoided in favor of recursion, where a function calls itself to iterate through collections (11:50-12:40).

what is functional reactive programming
Functional Reactive Programming (FRP) is a programming paradigm that merges two powerful concepts: functional programming and reactive programming.

It is designed to handle asynchronous event streams and time-varying values using a declarative style, making it particularly effective for complex systems like user interfaces, gaming, and robotics.

Core Pillars of FRP:
Reactive Programming: Focuses on data streams (often called Observables or Signals). Rather than handling events manually, the system reacts to sequences of events as they occur over time (e.g., user clicks, sensor input, or network responses).
Functional Programming: Emphasizes the use of pure functions, immutable data, and declarative transformations. You define what the data flow should be, rather than how to manage the state updates imperatively.
Key Concepts:
Data Streams: Asynchronous sequences of events that you can filter, map, and reduce.
Behaviors: Continuous, time-varying values that represent the current state at any given moment (e.g., current mouse position).
Declarative Dataflow: You define the relationship between inputs and outputs. When an input stream changes, the output automatically updates, reducing the risk of bugs caused by manual state management.
Functional Building Blocks: Operators like map, filter, and reduce are used to manipulate streams immutably, ensuring code is more predictable and easier to test.
Why Use FRP?
Simplifying Complexity: By treating events as streams, you can manage complex, asynchronous interactions with much cleaner, more readable code.
Consistency: The declarative approach minimizes side effects and state-related errors, leading to more stable applications.
Reusability: Pure functions and functional operators are modular, making them highly reusable across different parts of your system.
Common implementations include libraries like RxJS, ReactiveX, and languages like Elm.

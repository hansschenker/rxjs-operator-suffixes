# Four Scenarios, One Library — Deep Dive

**Section:** Composition & Concurrency
**Insight:** Four basic scenarios
**Lesson type:** Deep Dive
**Estimated duration:** 5 min

---

The four-scenario mental model is a diagnostic tool. Let's go through each scenario in depth, covering not just which operators belong there but why, and what the specific decision points are within each scenario.

Scenario one: you don't have an Observable. You need to create one. The question is what kind of source you're starting from. If you have a single static value or a small set of values, of emits each argument synchronously and completes. If you have an array, a Promise, or any iterable, from handles the conversion. If you're wrapping a DOM event, fromEvent takes an event target and an event name and gives you a push stream of events. If you need HTTP, ajax wraps an XMLHttpRequest. If you need a periodic timer, interval or timer handles it. And the special constants: EMPTY completes immediately without emitting, NEVER never emits or completes, throwError immediately errors. The choice of creation operator is a choice about what kind of source you're representing.

Scenario two: you have one Observable and want to transform it. This is the biggest family and the one developers learn first. Within this scenario there's still a meaningful set of decisions. Are you transforming each value independently? That's map. Are you accumulating state across values? That's scan, which is like Array's reduce but it emits the intermediate accumulate at each step — exactly what you want for state management. Are you filtering? filter passes values that satisfy a predicate. Are you controlling volume? take limits to N values, takeUntil stops on a signal, debounceTime suppresses values that come too quickly. Are you preventing redundant emissions of the same value? distinctUntilChanged holds the previous value and skips an emission if it equals the last one. Each of these answers a specific question about what you want to do with the values in a single stream.

Scenario three: you have multiple Observables and need to coordinate them. Here the decision is about the semantics of combination. merge subscribes to all source Observables concurrently and emits values as they arrive, with no ordering guarantee. concat subscribes to each source in sequence, only starting the next one when the previous completes — it preserves ordering at the cost of concurrency. combineLatest maintains one "latest value" slot per source and emits a combined tuple whenever any source emits, but only after every source has emitted at least once. zip pairs values by index — first from A with first from B, second from A with second from B — which is useful when you have correlated parallel streams. withLatestFrom is an asymmetric combination: it subscribes to a secondary Observable and samples its latest value whenever the primary Observable emits. And race subscribes to all sources but only forwards from whichever one emits first, abandoning the others — useful for timeout races. The key question in scenario three is always: what does it mean for multiple streams to be "combined"? The operators answer that question differently.

Scenario four: you have a stream of Observables and need to flatten it. This is the most consequential scenario because the choice between flattening operators has concrete correctness implications, not just performance ones. mergeMap subscribes to every inner Observable concurrently — all results interleaved in arrival order, no cancellation, no queuing. Use it when inner Observables are truly independent. concatMap subscribes to one inner at a time, queuing the rest — use it when order matters and you can't miss a value, like file writes or animation sequences. switchMap cancels the active inner when a new outer value arrives — use it when only the latest matters, like search typeahead or navigation. exhaustMap ignores new outer values while an inner is still active — use it to prevent overlap, like form submission or login.

The scenario model stops you browsing the operator list for something that sounds right. Instead you ask: which scenario am I in? Two questions instead of a hundred choices.
